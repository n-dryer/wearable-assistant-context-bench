Wearable Assistant Context Benchmark

A wearable or handheld coaching assistant sees and hears the user. The user swaps tools or walks into another room. The model needs to follow. This benchmark scores context tracking — keeping the model on the right frame as the user's situation changes mid-conversation.

Methodology

How the test works

Picture an assistant that sees what you see and hears what you say. You ask it questions and it answers.

In the middle of a conversation, your situation changes. You picked up a different tool. You walked into another room. The thing on screen switched. You didn't say it out loud. It just happened in front of the device. Then you ask a follow-up. Does the assistant answer about what's happening now, or about what was happening a minute ago?

Each scenario is structured around three turns (a turn is one user message plus the assistant's response). Turn 1 sets the scene. Between Turn 1 and Turn 2, the context shifts (something visible changes) without being announced. Turn 2 is a question that only makes sense if it noticed the shift. Turn 3 fires only on a Turn 2 miss: the user clarifies and the assistant gets one more chance, scored as the repair rate.

Each scenario carries a target label that says what the right answer should refer to:

  • current: the right answer is about what the camera shows in Turn 2 (the user's situation right now).
  • prior: the right answer is about an earlier scene (Turn 1, or a pre-conversation frame the user is referring back to).
  • clarify: the question is too ambiguous to answer from the available context; the assistant should ask which thing is meant.
  • abstain: the needed information isn't in the conversation; the assistant should decline rather than guess.

The 50 scenarios in the Scenario Bank are spread across eight kinds of context shift: object switched in hand, same object in a different state, the next step of a sequential task, a change of location, a new object brought into view, an absent referent (the thing the question is about is no longer visible), the content of a screen changing, and recall of something the device saw before the conversation began.

Heads up. The camera and audio inputs are text proxies (short scene descriptions and transcripts), not real video or raw audio. This keeps the benchmark cheap and reproducible, but a high score here doesn't guarantee one on real video or audio. See Out of scope.

Results

Six published runs

Each row below is an isolated experiment, not a ranked leaderboard. Higher score, more often the assistant tracks the shift. The primary score is balanced Turn 2 accuracy: the average of accuracy on current-target scenarios and accuracy on prior-target scenarios under the neutral system prompt, each given equal weight so the larger class doesn't dominate the headline. (v1 ships a neutral baseline prompt plus two nudge variants; only the baseline drives the primary score.)

Five runs use the Scenario Bank (50 scenarios); one uses a separate 20-scenario adversarial pack designed to discriminate at the top of the score range. CIs are 95% Wilson intervals per class and a 95% normal-approximation interval on the balanced mean.

Camera ablation

Strip the scene description and accuracy collapses

Same model, scene description shown vs hidden
60.6% → 14.4%

The camera ablation takes the same candidate model and the same judge and runs them twice: once with the scene description included in the prompt (baseline), once with it stripped (ablation-no-camera). Accuracy drops 46.2 percentage points. That rules out one alternative reading of the headline numbers, that the model is solving the task from question phrasing alone — it isn't, it needs the visual context to answer well. (This is about the text scene description we feed in; whether real video would behave the same is a separate question, called out in Out of scope. And on its own, the ablation isn't proof of deeper context tracking; the per-class pattern below fills in the rest.)

Per-class pattern

The model handles "current," but stumbles on "prior"

Cross-family integrity reference: Qwen3-VL-Plus + Gemini judge
100.0% on current / 8.3% on prior

When the right answer is about what the camera shows now (current), the model is strong; when the right answer requires referring back to an earlier scene (prior), it stumbles. Across all six runs the model is much better when the right answer is about the most recent frame than when the right answer is about an earlier frame. The baseline-qwen-cross-family run is the clearest example: 100% accuracy when the target is current, 8.3% when the target is prior. The model grounds in the latest visual input and struggles to refer back. Together with the camera ablation, this is the capability gap the benchmark targets: a strong read on what's in front of the model right now, paired with a weak read on what the user is referring back to.

Repair rate

When the user clarifies, can the model recover?

Deictic vs named repair on the Scenario Bank
Deictic 50/50 (100%) · Named 30/100 (30%)

If the model misses Turn 2, the user gets one clarifying follow-up. Recovery means the model gets it right after that follow-up; the repair rate is the fraction of misses that recover. v1 tests two ways the user might clarify.

The deictic anchor uses pointing-style language only: "no, this, what I'm holding now." (Deictic means words that point rather than name — "this", "that", "here", "now".) It mirrors how people actually clarify when they're holding something. The named anchor spells the objects out: "I mean the hammer I'm holding now, not the screwdriver from before." It is the most explicit clarification possible — if the model can't recover here, no clarification will save it.

The two scores cover different scenarios. The deictic anchor applies only when the referent is currently visible (50 misses, all 50 recovered). The named anchor covers the rest: misses on prior, clarify, and abstain targets where the user is referring back to something out of frame (100 misses, 30 recovered). The pattern: when the user can point, the model recovers every time. When the question is about something no longer visible, even explicit naming recovers only 30% of misses.

Judge agreement

Two models, the same labels: how often do they agree?

Cross-LLM agreement on the adversarial pack
Cohen's κ = 0.443 (moderate)

Each Turn 2 answer is read by a second model (the judge) that emits one of the four target labels. The default is cross-family judging: the judge comes from a different model maker than the candidate, so a model isn't grading itself. The fixed ranking judge is a second judge held constant across runs so candidates can be compared directly. The two judges read the same 300 adversarial trials. They agreed on 190 (Cohen's κ = 0.443, moderate). Not idiosyncratic to one family, but not lined up either. v1 reports this cross-LLM agreement as a substitute for human inter-annotator agreement (two human raters labeling the same trials, then comparing — planned future work).

Reading the gaps

When is one model's score meaningfully better than another's?

At the v1 sample size (50 scenarios × 5 trials = 250 paired observations under baseline), the minimum detectable effect at 80% power is approximately 13 percentage points. Two runs less than ~13 pp apart are not reliably distinguishable from sampling noise. The 17.1-point Gemini Flash vs Flash-Lite gap clears that threshold; the 6.4-point gap between Gemini Flash-Lite and Qwen3-VL-Plus does not. The standard paired test for binary outcomes on the same items is McNemar's test: of the 250 paired observations between baseline and baseline-alt, 55 disagreed (40 favoring Flash, 15 favoring Flash-Lite), χ2 = 10.47, p = 0.0012. (χ2 is the test statistic; p = 0.0012 means under 1-in-800 odds these results would arise from sampling noise alone.) Bigger sibling wins decisively on this bank. scripts/analyze_runs.py regenerates these numbers from the published JSONL transcripts.

One pattern the analysis surfaces is a ceiling effect (candidates score near the top of the range, leaving little room to discriminate between them): of the 50 scenarios, most pass at >= 80% on the cross-family integrity-reference run, and a smaller cluster fails at < 40%. Author difficulty (the tier the scenario was labeled with at authoring time) and empirical difficulty (the difficulty the cross-family run actually shows) agree on only 18 of 50 (36%). That motivates the 20-scenario adversarial pack and a 15-scenario harder-by-construction ceiling-test set (scenarios_v2_candidates.json) that fill cells the Scenario Bank under-covers. Re-running empirical difficulty across multiple cross-family candidates would tighten these labels; v1 reports the single available cross-family run.

Caveats on v1 numbers

What to keep in mind when reading the table

Mid-run, the API budget for non-Gemini providers ran out, leaving Gemini-direct as the only working path. Four of the five Gemini runs on the Scenario Bank ended up same-family (Gemini-Flash-Lite judging itself or Gemini-Flash) — pairings that can show self-preference bias, so those numbers may be inflated. baseline-qwen-cross-family is the cross-family reference for the Scenario Bank: same scenarios, same scoring rules, but a non-Gemini candidate paired with the Gemini judge. The 6.4-point gap between same-family baseline (60.6%) and cross-family baseline-qwen-cross-family (54.2%) is the visible self-preference signal, though candidate quality differs between the two runs and explains some of the gap. The gap also sits below the ~13-point minimum detectable effect at this sample size.

Limitations

What is out of scope

Inputs

  • Real video frames. The camera input is text descriptions of the scene, not actual frames. A model that does well here might not do well on real video.
  • Raw audio. Scoring is on text. The user's spoken turns are represented as text transcripts. Acoustic grounding (using audio properties of speech to identify what the user is referring to), speaker attribution (knowing which voice belongs to whom), and ambient audio cues (background sounds that disambiguate the scene) are not exercised.
  • Live audio and voice mode. A real wearable also needs live audio I/O with real-time interruption handling. None of that is exercised here.

Answer characteristics

  • Advice quality. The judge doesn't check whether the answer is correct, safe, or appropriate to the domain. A confidently wrong answer can pass.
  • Domain depth. Whether the advice is expert-level in any one activity covered (cooking, woodworking, music, etc.).
  • Proactive coaching. Whether the assistant volunteers help unprompted.

Conversation shape

  • Multi-turn flow past Turn 2. Whether the conversation works naturally from Turn 3 onward.
  • Long-horizon memory. Recall across days or weeks.

Engineering and statistics

  • Latency and cost. Wall-clock response time and price per call. Not measured.
  • Generalization beyond 5 trials per cell. v1 reports 95% CIs. Higher trial counts and seed sweeps are future work.
  • Human inter-annotator agreement. v1 reports cross-LLM agreement only. A second human rater on a 25% sample is the highest-priority future follow-up.
